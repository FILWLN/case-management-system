# 案管系统 - 待确认问题清单（给客户）

## 一、执恢业务流程（Q28-Q30）【高优先级】

### Q28: 执行恢复（执恢）的触发条件是什么？
**背景**：终本裁定后可能需要恢复执行
**需要确认**：
- [ ] 什么情况下需要执恢？（如发现新财产、被执行人恢复履行能力等）
- [ ] 执恢案件是**独立管理**还是作为**原执行案件的子案件**？
- [ ] 执恢是否需要重新立案？还是延续原案号？

**建议方案**：
```
方案A：独立管理
- 每次执恢生成新案件
- 关联原执行案件（通过parentCaseId）
- 单独编号：执恢-2024-001

方案B：子案件模式
- 在原执行案件下挂接执恢记录
- 不生成新案件，只更新状态和金额
- 历史以时间线形式展示
```

- [ ] 恢复执行需要**什么材料**？
- [ ] 恢复执行后，**还款数据如何计算**？（延续原还款计划/重新生成）
- [ ] 多次终本-恢复如何**统计报表**？（算1个案件还是多个）

## 二、状态流转规则（Q38-Q40）【高优先级】

### Q38: 状态变更方式？
**背景**：案件状态需要流转（网立→审核→立案→审理→执行→结案）
**需要确认**：
- [ ] **手动选择**：用户手动点击按钮切换状态？
- [ ] **自动推进**：满足条件自动流转？（如缴费后自动到"已受理"）
- [ ] **混合模式**：部分自动、部分手动？

**建议方案**：混合模式
```
自动流转：
- 网立提交 → 系统自动标记"网立立案已提交"
- 法院审核通过 → 系统自动标记"网立受理立案"

手动流转：
- 缴费后 → 用户手动确认"缴费通知"
- 收到传票 → 用户手动确认"传票已收到"
- 调解结果 → 用户手动选择"调解成功/失败"
```
